おすすめは「会社がOrganizationを契約し、各メンバーの個人GitHubアカウントを招待する」形です。会社資産はOrganizationに集約し、権限と2FAで管理します。

1. まず押さえる結論

今回の推奨は、TOKON用のGitHub Organizationを作り、各メンバーの個人GitHubアカウントを招待する方式です。

これはGitHubの標準的なチーム利用に近く、初期導入の負荷が低く、会社資産の置き場と権限管理を明確にできます。

おすすめ

GitHub Team / 初期導入向け

会社が有料Organizationを契約し、社員・協力者は自分のGitHubアカウントで参加します。

  • 導入しやすい
  • 一般的な運用に近い
  • Organization側で権限管理できる
  • 2FA必須化ができる

会社管理アカウント方式

GitHub Enterprise Managed Users / 大企業向け

会社のID基盤からGitHubユーザーを発行・管理する方式です。

  • 管理統制は強い
  • 導入・設定が重い
  • 情シス運用が前提
  • 初期導入では過剰になりやすい

Organization = 会社の共有スペース個人GitHub = 本人のログインID2FA = 二要素認証Repository = 案件・資料の置き場

5. リスクと対策

リスク個人アカウントのセキュリティに依存する

パスワード管理、2FA、SSHキー、連携アプリが甘いと、個人アカウント経由でOrganizationにアクセスされる可能性があります。

対策Organizationで2FAを必須化し、不要な権限・不要な連携を制限します。

リスク退職・外注終了時の外し忘れ

一時参加者を残したままにすると、不要なアクセス権が残ります。

対策参加者一覧を定期棚卸しし、外部協力者は必要最小限のリポジトリだけに招待します。

リスク会社資産が個人側に散らばる

個人リポジトリに本番コードや資料を置くと、資産の所在と責任が曖昧になります。

対策会社案件は必ずTOKON Organization配下に作成するルールを明文化します。

リスク管理者が一人だけになる

Organization ownerが一人だと、退職・休職・アカウント停止時に運用が詰まります。

対策ownerは最低2名。管理者権限は少数に限定します。

7. 導入ステップ

1

2
Organization作成GitHub TeamでTOKON用Organizationを作る。
3

4

8. 最低限の運用ルール案

会社案件・顧客案件・社内ツールは、原則としてTOKON Organization配下に作成する。
個人アカウントのprivate repositoryに会社資産を保管しない。
Organization ownerは最低2名とし、管理者権限をむやみに増やさない。
Organization参加者には2FAを必須化する。
外注・一時協力者はoutside collaboratorとして、必要なリポジトリだけに招待する。
退職・異動・契約終了時は、Organization/Repository権限を削除する。

9. FAQ

Q. 個人無料アカウントで会社の有料Organizationに入るのは変？

A. 変ではありません。GitHubでは一般的な使い方です。重要なのは、会社資産を個人リポジトリではなくOrganization配下に置くことです。

Q. 会社がGitHubアカウントを作って配るべき？

A. 初期導入では通常、各人が自分の個人アカウントを作成します。会社がアカウント自体を管理したい場合はEnterprise領域です。
