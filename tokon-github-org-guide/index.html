GitHub Organization 導入説明資料

TOKONでGitHubをチーム利用するための基本方針

「会社が有料の作業スペースを契約し、各メンバーは自分のGitHubアカウントで参加する」運用を基本に、所有権・権限・セキュリティを整理するための社内説明ガイドです。

対象：上長説明／導入判断／初期運用想定プラン：GitHub Team作成日：2026年5月14日

1. まず押さえる結論

推奨：TOKON用のGitHub Organizationを作り、各メンバーの個人GitHubアカウントを招待する。

GitHubでは、各利用者が「User account」でサインインし、会社やチームの共有スペースである「Organization」に参加して共同作業する形が標準的です。会社が管理すべきなのは、個人アカウントそのものではなく、Organization内の権限・リポジトリ・セキュリティ設定です。

通常運用

標準会社がGitHub Organizationを契約し、社員・協力者は自分のGitHub個人アカウントで参加します。

個人アカウントは本人のID。会社のコードや資料はOrganization配下のリポジトリに置きます。

大企業向け運用

EnterpriseGitHub Enterprise Managed Usersを使うと、IdPから会社管理のユーザーを発行・管理できます。

ただし、初期導入では設定・管理負荷が高く、まずはGitHub Teamから始めるのが現実的です。

2. なぜ個人アカウントで参加するのか

GitHubの基本設計

GitHubでは「人」はUser accountでログインします。Organizationは複数人が共有作業を行うための入れ物です。

  • 人：User account
  • 会社・部署・プロジェクト：Organization
  • 成果物・コード・HTML・設定：Repository

会社が制御する範囲

会社は個人アカウント自体を所有するのではなく、Organizationへの参加・権限・リポジトリを管理します。

  • Organizationへの招待／削除
  • チーム単位の権限付与
  • リポジトリ単位のアクセス制御
  • 2要素認証の必須化

3. 個人利用と法人利用の違い

項目個人アカウントに置く場合Organizationに置く場合
所有権個人に紐づく会社・チームの共有資産として扱いやすい
権限管理collaborator中心で簡易チーム・役割・リポジトリ単位で管理しやすい
退職・異動時資産移管が曖昧になりやすいメンバー削除・権限変更で対応しやすい
セキュリティ個人設定への依存が強いOrganizationポリシーを適用しやすい
推奨用途個人開発・試作会社案件・顧客案件・継続開発

4. リスクと対策

リスク個人アカウントのセキュリティに依存する

パスワード管理、2FA、SSHキー、連携アプリが甘いと、個人アカウント経由でOrganizationにアクセスされる可能性があります。

対策Organizationで2FAを必須化し、不要な権限・不要な連携を制限します。

リスク退職・外注終了時の外し忘れ

一時参加者を残したままにすると、後から不要なアクセス権が残ります。

対策参加者一覧を定期棚卸しし、外部協力者は必要最小限のリポジトリだけに招待します。

リスク会社資産が個人側に散らばる

個人リポジトリに本番コードや資料を置くと、資産の所在と責任が曖昧になります。なお、個人が自分のアカウント上にリポジトリを作る行為そのものを、通常のOrganization運用だけで完全に禁止するものではありません。

対策会社案件は必ずTOKON Organization配下に作成するルールを明文化し、正式な保管場所を一本化します。

リスク管理者が一人だけになる

Organization ownerが一人だと、退職・休職・アカウント停止時に運用が詰まります。

対策ownerは最低2名。管理者権限は少数に限定します。

6. 導入ステップ

1
方針承認会社資産はOrganization配下に置く、と決める。
2
Organization作成GitHub TeamでTOKON用Organizationを作る。
3
管理者設定ownerを最低2名にし、請求・管理担当を明確にする。
4
セキュリティ設定2FA必須、権限最小化、外部協力者ルールを設定。
5
運用開始案件・用途ごとにリポジトリを作り、棚卸しを定例化。

7. 最低限の運用ルール案

会社案件・顧客案件・社内ツールは、原則としてTOKON Organization配下に作成する。
個人アカウントのprivate repositoryに会社資産を保管しない。
Organization ownerは最低2名とし、管理者権限をむやみに増やさない。
Organization参加者には2FAを必須化する。
外注・一時協力者はoutside collaboratorとして、必要なリポジトリだけに招待する。
退職・異動・契約終了時は、Organization/Repository権限を削除する。
本番反映に関わるリポジトリは、直接更新ではなくpull requestとレビューを基本にする。
月1回程度、参加者・権限・不要リポジトリの棚卸しを行う。

8. FAQ

Q. 個人無料アカウントで会社の有料Organizationに入るのは変？

A. 変ではありません。GitHubでは一般的な使い方です。重要なのは、会社資産を個人リポジトリではなくOrganization配下に置くことです。

Q. 会社がGitHubアカウントを作って配るべき？

A. 通常は各人が自分の個人アカウントを作成します。会社がユーザーアカウントのライフサイクルまで管理したい場合は、Enterprise Managed Usersの検討領域です。

Q. 外注や一時協力者はどうする？

A. Organization memberにするより、outside collaboratorとして必要なリポジトリだけに招待する方が管理しやすいです。

Q. Copilotも同時に使える？

A. GitHub TeamとGitHub Copilotは別契約です。AIコード支援まで法人管理したい場合は、Copilot Business/Enterpriseの検討が必要です。

10. 初心者向け用語集

GitHubを初めて使う人が、資料内の言葉をざっくり理解するための説明です。厳密な技術定義よりも、社内説明で迷わないことを優先しています。

用語読み方意味社内での理解
GitHubギットハブコードや制作物の変更履歴を管理し、チームで共同作業するためのサービス。HTML、GAS、Webアプリ、設定ファイルなどを安全に保管・共同編集する場所。
Gitギットファイルの変更履歴を記録する仕組み。「誰が・いつ・何を変更したか」を追えるようにする土台。
AccountアカウントGitHubにログインするための個人ID。各メンバーが自分で作成し、会社のOrganizationに参加する。
Organizationオーガナイゼーション会社・チーム用の共有スペース。TOKONとして管理する正式な置き場。会社資産はここに集約する。
Repositoryリポジトリ／レポジトリコードや資料一式を入れる箱。変更履歴も一緒に管理される。案件・ツール・資料ごとの保管フォルダのようなもの。
Private repositoryプライベート・リポジトリ許可された人だけが見られる非公開リポジトリ。顧客案件や社内ツールは原則privateで管理する。
Public repositoryパブリック・リポジトリインターネット上の誰でも見られる公開リポジトリ。社内・顧客情報を含むものはpublicにしない。
MemberメンバーOrganizationに所属する利用者。継続的に関わる社員・担当者。
Outside collaboratorアウトサイド・コラボレーターOrganization全体には所属せず、特定リポジトリだけに招待される外部協力者。外注・一時協力者・限定的な確認者に使う。
OwnerオーナーOrganization全体を管理できる強い権限。最低2名にする。ただし増やしすぎない。
Read / Write / Adminリード／ライト／アドミンリポジトリに対する権限レベル。Readは閲覧、Writeは編集、Adminは管理。必要な人に必要な権限だけ付ける。迷ったら小さく始める。
2FAツーエフエー／二要素認証パスワードに加えて、スマホアプリや認証コードなどでも本人確認する仕組み。TOKON Organizationに参加する人は必須にする。未設定の人はアクセスできない運用にする。
CloneクローンGitHub上のリポジトリを自分のPCにコピーすること。開発作業ではよく使う。会社資産の扱いは社内ルールに従う。
Commitコミット変更内容を履歴として保存すること。「この修正を記録する」操作。
Pull requestプルリクエスト自分の変更を本体に取り込んでもらうための依頼。重要なリポジトリでは、いきなり本番反映せずレビューしてから取り込む。
Reviewレビュー変更内容を他の人が確認すること。ミスや不要な公開を防ぐための確認手順。
Branchブランチ本体とは別に作業するための分岐。本番用ファイルを直接触らず、安全に修正するための作業場所。
Forkフォークリポジトリを別の場所に複製して、自分側で独立して扱えるようにすること。会社のprivate repositoryでは、不要な複製を避けるため基本的に禁止・制限する。
SSH key / Tokenエスエスエイチキー／トークンGitHubへ安全に接続するための認証情報。パスワードのように扱う。共有しない、不要になったら削除する。
GitHub Teamギットハブ・チームOrganizationでチーム利用するための有料プラン。初期導入で想定する現実的な法人プラン。
GitHub Enterpriseギットハブ・エンタープライズ大企業向けの上位プラン。高度なID管理・監査・セキュリティ機能を使える。初期導入では必須ではない。将来、統制要件が強くなったら検討する。
